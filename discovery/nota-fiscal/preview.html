Premissas não-negociáveis

  • Toda regra fiscal vive no backend — frontend é só UX. CID/diagnóstico nunca aparece na descrição da NF (LGPD).
  • Multi-tenant por design — cada estabelecimento tem CNPJ, certificado A1, série, código de serviço próprios. Nada é "do Imedto".
  • Abstração via interface (INfsEmissaoGateway) — trocar de provedor é nova classe, não reescrita.
  • Outbox pattern para chamada externa — UI confirma rápido, worker garante idempotência e retry.
  • Custódia do certificado A1 delegada ao gateway — reduz superfície de ataque e responsabilidade do Imedto.
Como funciona

Fluxo de emissão (do agendamento à NF autorizada)

Sequência ponta-a-ponta. A clínica não vê o gateway externo — para ela, o Imedto é quem emite a NF. Tudo que envolve transmissão é assíncrono.

Caminho feliz

Rascunho → Em Transmissão → Autorizada. Tempo médio esperado: 2-15 segundos.

Rejeição

Erro de schema, código de serviço inválido, ISS divergente. UI mostra motivo + botão "Corrigir e reemitir". Lançamento volta para pendente.

Cancelamento

Backend valida prazo legal (varia por município, ~24h sem ônus) antes de chamar gateway. Fora do prazo → 422.

Arquitetura

Bounded context novo · Faturamento

Domínio próprio. Não mistura com Agendamento ou Financeiro. Segue DDD + CQRS já estabelecido no Imedto.

Agregados e eventos

  1. 1
    EmissorFiscal Configuração por estabelecimento — CNPJ, regime, certificado_id, série RPS, código de serviço padrão, alíquota ISS, gateway.
  2. 2
    LançamentoFaturável Origem: agendamento concluído + pago. Estado: pendente → faturado / dispensado. Único candidato a virar NF.
  3. 3
    NotaFiscal (aggregate raiz) Estado-máquina: Rascunho → EmTransmissão → Autorizada / Rejeitada / Cancelada. 1:1 com lançamento. Idempotência travada por lancamento_id.
  4. 4
    Domain events NotaFiscalSolicitada (handler chama gateway), NotaFiscalAutorizada (notifica paciente), NotaFiscalRejeitada, NotaFiscalCancelada.
  5. 5
    Outbox + worker UI confirma rápido (202). Worker assíncrono dispara o gateway, lida com retry, circuit breaker, DLQ. Webhook do gateway atualiza status.

Riscos críticos a validar

Cada item abaixo precisa de confirmação direta com o vendor ou de pesquisa específica. Nenhum bloqueia o discovery, mas alguns bloqueiam a assinatura de contrato.

A
Custódia do certificado A1 no Focus NFe
Confirmar se o gateway armazena o .pfx server-side ou se o Imedto precisa repassar a cada requisição. Afeta arquitetura de segredos e responsabilidade legal.
A
Reforma Tributária IBS/CBS — road map de cada vendor
NFS-e Nacional terá nova NT (NT-004/005 em piloto desde dez/2025). Confirmar plano de adequação antes de assinar contrato.
M
Cobertura SP híbrida (SNNFS-e ↔ Nota Fiscal Paulistana)
Validar comportamento do gateway na transição. SP é majoritário no volume esperado.
M
Tabela de preços do PlugNotas e Nota Gateway
Exigir proposta numérica escrita para 100/1k/10k NF/mês. Preço opaco é risco contratual e impede comparação.
M
SLA formal do Nota Gateway
Solicitar contrato com SLA numérico + histórico de uptime últimos 12 meses (rebranding em 2024).
B
SDK .NET no NFE.io
Repo GitHub tem POC em C# mas sem SDK oficial mantido. Confirmar com suporte ou usar HttpClient direto.
B
Webhook + idempotência via external_reference
Confirmar formato HMAC, retry policy e suporte a chave de idempotência por requisição em todos os candidatos.

Próximos passos

  1. Validação com 3-5 clientes-âncora — como emitem hoje? Provedor? Manual? Contador? Quanto pagam?
  2. Levantar municípios da base via query no banco — distinct(cidade) dos estabelecimentos ativos, ordenado por volume.
  3. POC com Focus NFe em homologação — emitir 5-10 NFs no município com maior volume.
  4. Call comercial Nota Gateway em paralelo — pedir proposta + lista de saúde.
  5. Definição de produto — opt-in, configuração mínima, política de descrição (LGPD), automação on/off.
  6. ADR formal após POC fixando gateway, modelo de custódia e abstração INfsEmissaoGateway.
