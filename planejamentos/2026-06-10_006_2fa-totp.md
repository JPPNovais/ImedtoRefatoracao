**Autor**: imedto-business-analyst
**Estimativa de esforço**: M
**Áreas regressivas tocadas**: autenticação (login/refresh/cookies BFF), permissionamento (toggle por estabelecimento para papel Dono), `MinhaContaView`. Não toca: prontuário, orçamento, financeiro, estoque, agenda.

> Roadmap: [`Docs/Roadmap/FASE_1_COMPLETUDE.md`](../Docs/Roadmap/FASE_1_COMPLETUDE.md) item **1.7**. Diferencial de segurança vendável para dado de saúde + hardening barato. SMS explicitamente fora (custo/complexidade).

## 1. Contexto e motivação

O sistema guarda dado pessoal sensível de saúde (Art. 11 LGPD). Hoje a única barreira de acesso é e-mail + senha. Uma credencial vazada (reuso de senha, phishing) dá acesso total ao prontuário de todos os pacientes de um estabelecimento. 2FA é a contramedida padrão de mercado e **commodity obrigatória** para argumentar segurança no plano Consultório (roadmap §pricing).

A escolha por **TOTP (RFC 6238)** é deliberada: é o segundo fator que **não tem custo de serviço externo** (sem SMS, sem provedor de OTP). O usuário instala um app autenticador (Google Authenticator, Authy, 1Password, etc.), escaneia um QR e passa a gerar códigos de 6 dígitos a cada 30s, offline. O backend valida o código contra o segredo compartilhado armazenado cifrado.

**Estado atual da auth (investigado)**:
- BFF de autenticação em `AuthController` (`backend/src/Services/Imedto.Backend.API/Controllers/AuthController.cs`). Tokens vivem em cookies HttpOnly (`access-token`, `refresh-token`); o front nunca os recebe.
- `IAuthService` / `LocalJwtAuthService` orquestram login/refresh/logout. `LoginAsync(email, senha)` retorna `AuthResult` (access + refresh + user). `ValidarSenhaAsync(usuarioId, senha)` já existe — valida senha **sem criar sessão** (hoje usado na anonimização da conta). É exatamente a primitiva que o segundo passo de login e o desativar-2FA precisam.
- Credencial em `AuthCredencial` (`Domain/Auth/AuthCredencial.cs`), 1:1 com `usuarios.id`. Já tem lockout por tentativas (`MaxTentativasFalhas = 10` → `Bloquear`), `TentativasFalhas`, `RegistrarFalhaLogin`.
- Rate limiting (sliding window por IP) em `Program.cs`: `auth-login` = 5 req, `auth-sensitive` = 3 req, `auth-refresh` = 10 req.
- Segredo cifrado em repouso: padrão canônico do projeto é `IDataProtectionProvider.CreateProtector("<purpose>")` → `.Protect()` / `.Unprotect()`. Já usado para o refresh token do certificado BirdID (`VincularCertificadoCommandHandler`, purpose `"assinatura.refresh_token"`). Tabela `assinatura_certificados.refresh_token` é `text` guardando o valor protegido. **O segredo TOTP segue exatamente esse padrão.**
- `MinhaContaView.vue` já tem um card **"Segurança"** com `AlterarSenhaModal`. A seção de 2FA entra nesse card.
- Config de estabelecimento mora no aggregate `Estabelecimento` (`Domain/Estabelecimentos/Estabelecimento.cs`) — colunas diretas (não tabela de settings separada). O toggle "exigir 2FA do Dono" é uma coluna booleana nova ali.

## 2. Persona-alvo

- **Qualquer usuário** (dono, admin, recepção, profissional, financeiro) que queira proteger a própria conta — ativação **opcional e individual** em Minha Conta.
- **Dono** que, como responsável pelo tenant, quer **forçar 2FA para o papel Dono** do seu estabelecimento (toggle de configuração) — endurecimento da conta de maior privilégio.
- Momento da jornada: configuração de conta (uma vez) + todo login subsequente (segundo passo).
- Frequência: ativação rara; segundo passo a cada login (até a sessão durar).

## 3. Escopo

**Inclui**:
- **Ativação opcional por usuário** em Minha Conta → card "Segurança" → "Ativar verificação em duas etapas":
  - Backend gera segredo TOTP, devolve a `otpauth://` URI (para QR) + o segredo em base32 (fallback de digitação manual).
  - Front renderiza o QR a partir da URI.
  - Usuário confirma digitando um código TOTP válido. Só então o 2FA fica **ativo** e os **10 códigos de recuperação** são gerados e exibidos **uma única vez**.
- **Desativação** em Minha Conta: exige **senha atual válida E um código TOTP válido** (ou código de recuperação). Remove segredo + códigos de recuperação.
- **Login em dois passos**: após senha válida, se o usuário tem 2FA ativo, o primeiro passo **NÃO** seta os cookies de sessão; retorna um estado "requer segundo fator". O segundo passo recebe um código TOTP (ou código de recuperação) e, se válido, seta os cookies (mesmo `AuthResult` de hoje).
- **10 códigos de recuperação one-time** gerados na ativação. Armazenados como **hash** (nunca em claro). Cada um vale uma vez; consumido marca como usado.
- **Config por estabelecimento**: toggle "Exigir 2FA para o papel Dono" (default **desligado**). Quando ligado, um Dono sem 2FA ativo é **forçado a configurar** no próximo login naquele contexto (gate de fluxo, não bloqueio cego).
- **Auditoria**: ativação, desativação e uso de código de recuperação registrados (sem PII).
- **Rate limit no segundo passo** de login (mesmo padrão `auth-sensitive` = 3 req por IP).
- **Schema muda** → aciona `imedto-database`.

**Não inclui** (explicitamente fora):
- **SMS** OTP, **e-mail** OTP, **WebAuthn**/passkeys/chave física.
- **"Lembrar este dispositivo"** (pular 2FA por N dias num device confiável). Fica como evolução futura — registrado em §8.
- 2FA para o **admin global** (`/api/admin/*`). O admin global tem fluxo de auth próprio (`AdminAuthController`, `adminAuthStore`); esta demanda é só para usuários de tenant. Backlog separado se solicitado.
- Forçar 2FA para outros papéis além de Dono (recepção, profissional). MVP só tem o toggle para Dono.
- Regeneração de códigos de recuperação numa tela dedicada após a ativação (o usuário que perder os códigos e ainda tiver o app pode desativar+reativar). Backlog se solicitado.

## 4. Regras de negócio

> **Premissa global (CLAUDE.md)**: toda regra abaixo é **fonte de verdade no backend**. O front é UX. Nenhuma validação de TOTP/recuperação acontece só no front.

- **R1 — Geração do segredo (ativação, passo 1)**. Endpoint autenticado gera um segredo TOTP aleatório (mín. 160 bits / 20 bytes, base32), **ainda não ativa** o 2FA, e devolve `{ otpauthUri, segredoBase32 }`. O segredo é guardado em estado **"pendente"** (cifrado) até a confirmação. `otpauthUri` no formato `otpauth://totp/Imedto:{email}?secret={base32}&issuer=Imedto&algorithm=SHA1&digits=6&period=30`. Mora em: **Handler + Domain** (a entidade de 2FA encapsula a transição pendente→ativo). Validado em: backend.
- **R2 — Confirmação da ativação (passo 2)**. Endpoint autenticado recebe um código de 6 dígitos. Valida contra o segredo pendente com janela de **±1 step** (tolerância de clock, RFC 6238). Se válido: marca 2FA como **ativo**, gera 10 códigos de recuperação, persiste os **hashes**, e retorna os 10 códigos em claro **uma única vez**. Se inválido: 422 genérico, 2FA permanece inativo. Mora em: **Handler + Domain**. Validado em: backend.
- **R3 — Login passo 1 (senha)**. `POST /api/auth/login` continua validando e-mail+senha. **Mudança**: se a credencial autenticada tem 2FA ativo, **não** seta cookies; retorna `{ requerSegundoFator: true }` (200) + um identificador de desafio efêmero seguro (ver §5 — não expor `usuario_id` em claro). Se 2FA inativo: comportamento atual inalterado (seta cookies). Mora em: **AuthController + IAuthService**. Validado em: backend.
- **R4 — Login passo 2 (código)**. `POST /api/auth/login/2fa` recebe o desafio + um código. Aceita **código TOTP válido** (janela ±1 step) **OU** um **código de recuperação** ainda não usado. Em sucesso: seta cookies (mesmo `AuthResult`/`SetAuthCookies` de hoje), e — se foi código de recuperação — **marca aquele código como usado**. Em falha: 422 genérico ("Código inválido."). Mora em: **AuthController + IAuthService + Domain**. Validado em: backend. **Rate limit `auth-sensitive` (3 req/IP)** no endpoint.
- **R5 — Anti-bypass do segundo passo**. O desafio do passo 1 é a **única** forma de chegar ao passo 2. Não pode existir caminho que emita cookies para um usuário com 2FA ativo sem passar pelo passo 2. O desafio expira em **5 minutos** e é de uso único. Falha-fechada: desafio inválido/expirado → 422 genérico, sem revelar se o usuário existe. Mora em: **IAuthService (emissão/validação do desafio)**. Validado em: backend.
- **R6 — Desativação**. `POST /api/auth/2fa/desativar` (autenticado) exige **senha atual válida** (`ValidarSenhaAsync`) **E** um código TOTP válido **ou** código de recuperação válido. Em sucesso: remove segredo + todos os códigos de recuperação, marca 2FA inativo. Em falha de qualquer um dos dois fatores: 422 genérico, 2FA permanece ativo. Mora em: **AuthController + IAuthService + Domain**. Validado em: backend.
- **R7 — Códigos de recuperação são one-time e hasheados**. Nunca persistir código de recuperação em claro. Hash com o mesmo algoritmo de senha já usado no projeto (dev/db confirmam o hasher de `SenhaHash`; reusar, não inventar). Consumo é idempotente-seguro: um código usado nunca mais autentica. Mora em: **Domain + Handler**. Validado em: backend.
- **R8 — Segredo cifrado em repouso**. O segredo TOTP (pendente e ativo) é guardado **cifrado** via `IDataProtectionProvider.CreateProtector("auth.totp.secret")` → `.Protect()`, decifrado com `.Unprotect()` só no momento de validar um código. Espelha o padrão de `assinatura.refresh_token`. **Nunca** retornado em payload depois da ativação (o `otpauth`/base32 só aparece **durante** a ativação, antes da confirmação). Mora em: **Handler/Infra**. Validado em: backend.
- **R9 — Toggle por estabelecimento (exigir Dono)**. Config booleana `exigir_2fa_dono` no aggregate `Estabelecimento`, default `false`. Só o **Dono** pode alterá-la (RBAC). Mora em: **Domain (`Estabelecimento`) + Handler de atualização de config**. Validado em: backend + front (toggle só visível/acionável para Dono).
- **R10 — Enforcement do toggle no login do Dono**. Quando `exigir_2fa_dono = true` num estabelecimento e o usuário que loga é Dono **daquele** estabelecimento e **não** tem 2FA ativo, o sistema sinaliza que ele deve configurar 2FA. **Decisão (default mais simples)**: o login **conclui** (cookies setados) mas o front recebe a flag `deveConfigurar2fa: true` e **força** o usuário para a tela de configuração de 2FA em Minha Conta, bloqueando navegação para outras rotas do app até ativar (guard de rota, espelho do padrão de onboarding pendente). **Não** bloqueia o login em si — bloquear login deixaria o Dono fora do próprio sistema sem caminho de saída. O backend continua sendo fonte da verdade: enquanto o Dono não ativar, ações de domínio sensíveis **não** ganham proteção extra (o enforcement é de fluxo/UX para empurrar a ativação). Mora em: **decisão de produto + guard de rota (front) + flag no bootstrap/me (back)**. Ver §11 (premissa explícita — usuário pode trocar por "bloqueio duro" via addendum).
- **R11 — Multi-tenant do toggle**. `exigir_2fa_dono` é **por estabelecimento**. Um Dono com N estabelecimentos pode ter o toggle ligado em um e desligado em outro; o enforcement (R10) avalia o estabelecimento do contexto. A flag de 2FA do **usuário** (ativo/inativo) é **global** (vive na credencial, não por tenant) — 2FA protege a conta inteira, não um tenant. Mora em: **Domain**. Validado em: backend.
- **R12 — LGPD / sem PII**. Nenhum log estruturado ou mensagem de erro contém o segredo TOTP, código de recuperação, código TOTP digitado, e-mail, CPF ou telefone. Mensagens de erro de 2FA são **genéricas** ("Código inválido.", "Não foi possível ativar a verificação em duas etapas."). A auditoria (R13) registra apenas `usuario_id`, ação e timestamp. Mora em: **Handlers + AuthController**. Validado em: backend.
- **R13 — Auditoria das ações de 2FA**. Registrar em audit **ativação**, **desativação** e **uso de código de recuperação** (não registrar cada login bem-sucedido com TOTP — isso é login normal). Cada linha: `{ usuario_id, acao, ocorrido_em }`, sem PII. **Decisão**: reusar o mecanismo de audit já existente mais adequado a evento de conta (dev/db avaliam: o audit de paciente/prontuário NÃO serve — é PII clínica; o audit do admin global é de outro contexto). Se nenhum audit de "evento de conta de usuário de tenant" existir, criar uma tabela mínima `usuario_seguranca_audit` append-only (`id, usuario_id, acao, ocorrido_em, ip_origem?`) — segue o padrão append-only de `paciente_acesso_log`. Mora em: **Handler + nova tabela (se necessária)**. Validado em: backend. Ver §5 e §11.
- **R14 — Interação com troca de senha**. A troca de senha hoje revoga todas as sessões (já existe). 2FA **não** é afetado por troca de senha (segredo permanece). Não introduzir acoplamento novo. Mora em: **nenhuma mudança** — apenas garantir que `AlterarSenhaAsync` não mexe no 2FA. Validado em: backend (regressão).

## 5. Modelo de dados

> **SCHEMA MUDA → aciona `imedto-database`.** O DB agent é o único autor de migration. Toda migration multi-tenant-aware, idempotente, com índice dia 1.

**Onde mora o estado de 2FA do usuário** — duas opções (dev/db decidem a mais limpa; recomendação do BA abaixo):

- **Recomendação**: tabela nova `usuario_2fa` (1:1 opcional com `usuarios.id`), em vez de inflar `auth_credenciais`. Justificativa: o segredo e o ciclo pendente→ativo são um conceito coeso; manter `auth_credenciais` focada em senha/lockout. Colunas sugeridas:
  - `usuario_id uuid PK` (= `usuarios.id`).
  - `segredo_cifrado text NOT NULL` (valor protegido por `IDataProtector`).
  - `status text NOT NULL` (`Pendente` | `Ativo`) — `HasConversion<string>` no padrão das outras enums.
  - `ativado_em timestamptz NULL`.
  - `criado_em timestamptz NOT NULL`, `atualizado_em timestamptz NULL`.
- **Códigos de recuperação**: tabela `usuario_2fa_codigo_recuperacao` (1:N):
  - `id` (PK gerado), `usuario_id uuid NOT NULL` (FK lógica), `codigo_hash text NOT NULL`, `usado_em timestamptz NULL`, `criado_em timestamptz NOT NULL`.
  - Índice `(usuario_id)` para buscar os códigos do usuário no login passo 2.
- **Toggle por estabelecimento**: coluna nova `exigir_2fa_dono boolean NOT NULL DEFAULT false` em `public.estabelecimentos` (aggregate `Estabelecimento`). Migration idempotente `ADD COLUMN IF NOT EXISTS`.
- **Auditoria** (R13): se não houver tabela de evento de conta, criar `usuario_seguranca_audit` append-only: `id`, `usuario_id uuid NOT NULL`, `acao text NOT NULL` (`Ativou2fa` | `Desativou2fa` | `UsouCodigoRecuperacao`), `ocorrido_em timestamptz NOT NULL`, `ip_origem text NULL` (45 chars, padrão de `paciente_acesso_log`). Índice `(usuario_id, ocorrido_em)`.

**Segredo cifrado em repouso**: padrão de `assinatura_certificados.refresh_token` — coluna `text` guardando o output de `IDataProtector.Protect`. **Não** usar pgcrypto/coluna binária; usar Data Protection (chave gerenciada pelo runtime, consistente com o resto do projeto). Confirmar com `imedto-database` que o keyring do Data Protection em produção é persistente (senão o `.Unprotect` quebra após restart — risco de infra; ver §8).

**Desafio de login (R3/R5)**: efêmero, **não** precisa de tabela. Recomendação: token opaco assinado/cifrado (via `IDataProtector` purpose `"auth.totp.challenge"`) contendo `{ usuario_id, exp }`, devolvido ao front no passo 1 e reenviado no passo 2. Sem persistência → sem limpeza de lixo, sem leak de `usuario_id` em claro. (Alternativa: cache em memória com TTL 5min — mas o desafio precisa sobreviver entre dois requests possivelmente em nós diferentes; o token cifrado é mais robusto.) Dev/db decidem; o BA recomenda o token cifrado.

**Sem PII nas tabelas novas** — nenhuma guarda nome/CPF/telefone. Segredo e códigos sempre cifrados/hasheados.

## 6. UX e fluxo

**Minha Conta → card "Segurança"** (`MinhaContaView.vue`, abaixo do botão "Trocar senha"):
- Estado **inativo**: subtítulo "Adicione uma camada extra de proteção exigindo um código do seu aplicativo autenticador a cada login." + `AppButton` "Ativar verificação em duas etapas".
- Ao clicar → abre **modal de ativação** (componente novo `Ativar2faModal.vue`, espelhando o padrão de `AlterarSenhaModal.vue`):
  1. Passo "Escaneie": QR gerado da `otpauthUri` + segredo base32 copiável (fallback). Texto: "Escaneie com Google Authenticator, Authy, 1Password ou similar."
  2. Passo "Confirme": campo de 6 dígitos + `AppButton` "Confirmar e ativar". Erro genérico inline em código inválido.
  3. Passo "Guarde seus códigos de recuperação": lista dos 10 códigos + botão "Copiar" e "Baixar .txt". Aviso forte: "Guarde em local seguro. Eles não serão exibidos novamente. Cada código funciona uma única vez." Botão "Concluir" fecha o modal.
- Estado **ativo**: indicador "Verificação em duas etapas ativa" (estilo `cert-status--ok` já existente na view) + `AppButton` (variant secondary, ícone trash) "Desativar".
- Ao clicar em desativar → **modal de desativação**: campo senha atual + campo código (TOTP ou recuperação) + `AppButton` "Desativar" (variant danger). Erro genérico em falha.
- **QR**: usar uma lib leve de geração de QR no front (dev escolhe; reuso se já houver dependência de QR na base — `grep` antes). O backend devolve só a URI; o desenho do QR é UX no front.

**Login em dois passos** (`LoginView.vue`):
- Passo 1 inalterado visualmente (e-mail + senha). Em resposta `requerSegundoFator: true`, a view **transiciona** para um segundo painel: campo de 6 dígitos "Código de verificação" + link discreto "Usar um código de recuperação" (que troca o input para aceitar um código de recuperação). `AppButton` "Verificar".
- Erros: genéricos. Voltar ao passo 1 limpa o desafio.
- O desafio expira em 5 min → mensagem "Sua sessão de verificação expirou. Faça login novamente." e volta ao passo 1.

**Enforcement do Dono (R10)** — guard de rota:
- Espelhar o padrão de `onboardingPendente` em `frontend/src/router/index.ts`: se o bootstrap/me retorna `deveConfigurar2fa: true`, redirecionar para `MinhaConta` (com query/flag para abrir o modal de ativação e um banner explicativo "Seu estabelecimento exige verificação em duas etapas para o responsável. Ative para continuar."). Bloquear navegação para outras rotas `requiresTenant` até ativar.

**Toggle por estabelecimento** (config de Estabelecimento — `EstabelecimentoView.vue`, seção de segurança/geral; dev confirma a seção mais coerente no master-detail de configurações):
- `AppPillToggle` (componente do DS já existente, `components/ui/AppPillToggle.vue`) ou switch padrão: "Exigir verificação em duas etapas para o responsável (Dono)". Subtexto explicando o efeito. Visível/acionável **só para Dono**.

**Estados** (todas as telas):
- **Loading**: botões com `:loading` (padrão `AppButton`).
- **Erro**: mensagens **genéricas** (`msg-erro`), nunca revelando segredo/código/se o usuário existe.
- **Sucesso ativação**: a própria tela de códigos de recuperação é o sucesso; ao concluir, card volta para estado "ativo".
- **Vazio**: n/a (é fluxo, não lista).

**Tipografia**: respeitar CLAUDE.md §5 — sem `font-size`/`font-weight` literais; usar tokens / classes do DS (`ds-card-title`, `field-label`, etc.).

**Mobile-ready**: modais e o segundo passo de login funcionam em coluna única; campo de 6 dígitos com `inputmode="numeric"`.

## 7. Critérios de aceite (testáveis)

- **CA1 (ativação — caminho feliz)**: Dado um usuário sem 2FA, Quando ele clica "Ativar verificação em duas etapas", escaneia o QR e digita um código TOTP válido, Então o 2FA fica ativo, exatamente **10** códigos de recuperação são exibidos uma única vez, e o card passa a mostrar "Verificação em duas etapas ativa".
- **CA2 (ativação — código inválido)**: Dado o modal de ativação no passo de confirmação, Quando o usuário digita um código incorreto, Então recebe erro genérico ("Código inválido."), o 2FA **não** é ativado e nenhum código de recuperação é gerado.
- **CA3 (segredo cifrado em repouso)**: Dado um 2FA ativado, Quando se inspeciona a linha em `usuario_2fa`, Então `segredo_cifrado` **não** é o segredo em claro nem o base32 — é o output de `IDataProtector.Protect` (purpose `auth.totp.secret`); e o segredo só é decifrado em memória ao validar um código.
- **CA4 (login passo 1 sem 2FA — regressão)**: Dado um usuário **sem** 2FA, Quando faz login com senha correta, Então os cookies de sessão são setados normalmente (comportamento atual inalterado) e **nenhum** segundo passo é pedido.
- **CA5 (login passo 1 com 2FA)**: Dado um usuário **com** 2FA ativo, Quando faz login com senha correta, Então **nenhum cookie de sessão é setado**, a resposta é `{ requerSegundoFator: true }` (200) com um desafio efêmero, e o front transiciona para o painel de código.
- **CA6 (login passo 2 — TOTP válido)**: Dado o desafio do passo 1, Quando o usuário envia um código TOTP válido (janela ±1 step), Então os cookies de sessão são setados e ele entra no app.
- **CA7 (login passo 2 — código de recuperação)**: Dado o desafio do passo 1, Quando o usuário usa um código de recuperação **não usado**, Então a sessão é criada e aquele código é **marcado como usado**; reusar o mesmo código numa nova tentativa **falha** com erro genérico.
- **CA8 (login passo 2 — código inválido + rate limit)**: Dado o desafio do passo 1, Quando o usuário erra o código, Então recebe 422 genérico ("Código inválido."); e após exceder o limite da política `auth-sensitive` (3 req/IP) o endpoint do passo 2 responde 429.
- **CA9 (anti-bypass)**: Dado um usuário com 2FA ativo, Quando se tenta obter sessão sem passar pelo passo 2 (ex.: chamar diretamente um endpoint de sessão, ou reenviar só o passo 1), Então **nenhum** cookie de sessão válido é emitido; o único caminho de sucesso é passo 1 → passo 2.
- **CA10 (desafio expira)**: Dado um desafio emitido há mais de 5 minutos, Quando o usuário envia o código no passo 2, Então recebe erro genérico de expiração e deve refazer o login (passo 1).
- **CA11 (desativação — caminho feliz)**: Dado um usuário com 2FA ativo, Quando informa senha atual válida **e** um código válido (TOTP ou recuperação) e confirma, Então o 2FA é desativado, segredo e códigos de recuperação são removidos, e o próximo login não pede segundo fator.
- **CA12 (desativação — falta um dos fatores)**: Dado o modal de desativação, Quando o usuário informa senha correta mas código inválido (ou vice-versa), Então recebe 422 genérico e o 2FA **permanece ativo**.
- **CA13 (RBAC do toggle por estabelecimento)**: Dado um usuário com papel **Profissional/Recepção** (não Dono), Quando acessa a config do estabelecimento, Então o toggle "Exigir 2FA para o responsável" **não** aparece no front e, se a API for chamada diretamente, retorna 403/SemAcesso; apenas o **Dono** consegue alterá-lo.
- **CA14 (enforcement do Dono)**: Dado um estabelecimento com `exigir_2fa_dono = true` e um Dono **sem** 2FA, Quando esse Dono loga naquele contexto, Então ele é redirecionado para a tela de configuração de 2FA com banner explicativo e **não consegue navegar** para outras rotas do tenant até ativar; após ativar, a navegação é liberada.
- **CA15 (multi-tenant do toggle)**: Dado um Dono com dois estabelecimentos, um com `exigir_2fa_dono = true` e outro `false`, Quando ele alterna o contexto, Então o enforcement (R10) só dispara no estabelecimento com o toggle ligado; o toggle de um estabelecimento **não** afeta o outro.
- **CA16 (multi-tenant — escopo da credencial)**: Dado um usuário com 2FA ativo, Quando ele troca de estabelecimento, Então o estado de 2FA (ativo) é o **mesmo** em qualquer contexto — 2FA é da conta, não do tenant.
- **CA17 (LGPD — sem PII em log/erro)**: Dado qualquer erro ou sucesso em ativação/login-2fa/desativação, Quando se inspecionam logs e o corpo das respostas de erro, Então **não** aparecem segredo TOTP, código de recuperação, código digitado, e-mail, CPF ou telefone; as mensagens de erro são genéricas e não revelam se o usuário existe.
- **CA18 (LGPD — segredo não vaza pós-ativação)**: Dado um 2FA já ativo, Quando se chamam os endpoints de estado da conta (`/auth/me`, bootstrap, qualquer GET de perfil), Então **nenhum** deles devolve o segredo TOTP nem os códigos de recuperação; o segredo/base32 só transita **durante** o fluxo de ativação, antes da confirmação.
- **CA19 (auditoria)**: Dado que um usuário ativa, depois desativa, depois (em outra ativação) usa um código de recuperação no login, Então existem linhas de auditoria para `Ativou2fa`, `Desativou2fa` e `UsouCodigoRecuperacao`, cada uma com `{ usuario_id, ocorrido_em }` e **sem** PII; logins bem-sucedidos com TOTP **não** geram linha extra (são login normal).
- **CA20 (regressão — troca de senha)**: Dado um usuário com 2FA ativo, Quando ele troca a senha (fluxo atual), Então as sessões são revogadas como hoje **e** o 2FA permanece ativo (segredo intacto); o próximo login pede senha nova + segundo fator.
- **CA21 (códigos de recuperação são hash)**: Dado um 2FA ativado, Quando se inspeciona `usuario_2fa_codigo_recuperacao`, Então `codigo_hash` **não** é o código em claro; e validar um código no login compara por hash.

## 8. Riscos e dependências

- **Data Protection keyring em produção**: `IDataProtector.Unprotect` só funciona se o keyring for **persistente e estável** entre restarts/instâncias. Se hoje as chaves do Data Protection não estiverem persistidas (ex.: efêmeras por container), o segredo TOTP fica ilegível após restart → usuários travados no login. **Ação para `imedto-database`/infra**: confirmar a persistência do keyring (mesma dependência que o certificado BirdID já tem). Se não estiver persistido, é pré-requisito de infra — sinalizar ao orquestrador antes de codar. Atualizar `Docs/INFRA.md` se uma config de keyring precisar ser documentada.
- **Clock skew**: validar TOTP com janela **±1 step** (RFC 6238). Janela maior aumenta superfície de ataque; janela zero rejeita usuários com relógio levemente fora. Fixar em ±1.
- **Anti-bypass (R5/CA9)** é o ponto crítico de segurança: qualquer caminho que emita cookies para usuário-com-2FA sem o passo 2 é uma vulnerabilidade. QA deve testar explicitamente (chamar endpoints fora de ordem).
- **Área regressiva — login/refresh**: a mudança no `LoginAsync`/`AuthController.Login` é cirúrgica mas toca o caminho mais sensível do sistema. Toda a suíte de auth (`RateLimitTests`, testes de login) deve continuar verde. Refresh/logout não mudam.
- **Lockout × 2FA**: hoje o lockout conta falhas de **senha**. Falhas no **passo 2** (código) devem ter o seu próprio limite (rate limit `auth-sensitive`) — **não** incrementar `TentativasFalhas` da senha (senão um atacante tranca a conta da vítima só errando códigos). Confirmar separação.
- **Perda total de fatores**: usuário que perde o app **e** os códigos de recuperação fica sem acesso. No MVP, o caminho é o suporte/Dono — **não** há auto-recuperação. Registrar como limitação conhecida (e futura tela de "reset de 2FA pelo Dono", backlog).
- **Evolução futura — "lembrar dispositivo"**: fora do MVP. Quando entrar, exige cookie de device + tabela de devices confiáveis; planejar para não conflitar com a estrutura atual.
- **Dependência de QR no front**: confirmar/escolher lib de QR (reuso se já houver). O backend não desenha QR — só devolve a `otpauth` URI.

## 9. Observações para execução

**Não-negociável**:
- Segredo TOTP **cifrado em repouso** via `IDataProtector` purpose `auth.totp.secret` (espelho de `assinatura.refresh_token`) — R8/CA3.
- Códigos de recuperação **hasheados**, one-time — R7/CA21/CA7.
- Login em dois passos com **anti-bypass** real (R5/CA9) e **sem** setar cookies no passo 1 quando há 2FA (R3/CA5).
- Rate limit `auth-sensitive` no passo 2 (R4/CA8); **não** confundir com o lockout de senha (risco descrito em §8).
- Mensagens **genéricas**, sem PII, sem revelar existência de usuário (R12/CA17).
- Toggle por estabelecimento **só Dono** (R9/CA13) e **multi-tenant** (R11/CA15).
- Reuso de `ValidarSenhaAsync` no desativar (R6) e do `SetAuthCookies`/`AuthResult` no passo 2 (R4) — não duplicar fluxo de sessão.

**Liberdade técnica (dev/db decidem)**:
- Lib de TOTP no backend (escolher uma madura para .NET; **não** implementar HMAC-SHA1/base32 na mão se houver lib confiável — mas validar a escolha). Reuso se já houver algo no projeto (`grep`).
- Lib de QR no front.
- Estrutura exata das tabelas (recomendações em §5) — desde que cumpra cifragem/hash/multi-tenant/audit.
- Formato do desafio de login (token cifrado recomendado vs cache TTL) — desde que cumpra R5/CA9/CA10.
- Onde colocar o toggle na UI de configurações (seção mais coerente no master-detail).

**Reuso obrigatório (grep antes de criar)**:
- `IDataProtectionProvider` / `CreateProtector` (`VincularCertificadoCommandHandler`).
- `IAuthService.ValidarSenhaAsync`, `LoginAsync`, `AuthController.SetAuthCookies`/`AuthResult`.
- Hasher de senha do projeto (para os códigos de recuperação).
- `AlterarSenhaModal.vue` como template dos modais; `AppButton`, `AppPillToggle`, card "Segurança" de `MinhaContaView.vue`.
- Padrão de guard `onboardingPendente` em `router/index.ts` (para o enforcement do Dono).
- Políticas de rate limit em `Program.cs` (`auth-sensitive`).

**Aciona `imedto-database`**: **sim, obrigatório** — tabelas de 2FA + códigos de recuperação + coluna `exigir_2fa_dono` em `estabelecimentos` + (se necessário) `usuario_seguranca_audit`. Migration idempotente; confirmar persistência do keyring do Data Protection.

## 11. Premissas dos pontos abertos (resolvidas pelo BA; usuário pode corrigir via addendum antes do dev)

1. **Enforcement do Dono (R10)** = login **conclui** + flag `deveConfigurar2fa` + guard de rota que **força** a configuração antes de usar o app (espelho de onboarding pendente). **Não** é bloqueio duro do login. Justificativa: bloquear o login deixaria o Dono fora do próprio sistema sem caminho de saída. Se o usuário preferir bloqueio duro (negar sessão até o Dono ativar via outro canal), vira addendum.
2. **Auditoria (R13)** = reusar mecanismo de evento de conta existente; se não houver, criar `usuario_seguranca_audit` mínima. Login com TOTP **não** gera linha (é login normal); só ativação/desativação/uso-de-recuperação são auditados.
3. **Desafio de login (§5)** = token efêmero cifrado (`IDataProtector` purpose `auth.totp.challenge`), TTL 5 min, uso único — sem tabela. Recomendação do BA; dev/db podem trocar por cache TTL desde que cumpram CA9/CA10.
4. **Algoritmo/parâmetros TOTP** = SHA1, 6 dígitos, period 30s, janela ±1 step (defaults RFC 6238 universalmente compatíveis com apps autenticadores). Não expor configuração ao usuário no MVP.
5. **Quantidade de códigos de recuperação** = 10 (conforme decisão fornecida). Sem tela de regeneração no MVP (desativar+reativar gera novos).
6. **Estado de 2FA** = **global por conta** (não por tenant). O **toggle de exigência** é por estabelecimento. Distinção registrada em R11/CA16.
